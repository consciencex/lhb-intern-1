# Claude Design Brief — ② Automate-or-Not (decision simulation)

> วาง markdown นี้ลง Claude Design เพื่อสร้าง UI · ส่วน "สำหรับ Claude Code" อยู่ท้ายไฟล์ (แยกชัด)

## บริบท
เกมตัดสินใจสำหรับนักศึกษาฝึกงานธนาคาร ~30 คน ผู้เล่นสวมบทหัวหน้าทีม งานโผล่มาทีละสถานการณ์ (บางอย่าง ปริมาณสูง/เสี่ยงต่ำ = เหมาะ automate; บางอย่าง เสี่ยงสูง/ถูกกำกับ = ต้องมีคน) แต่ละข้อเลือก **Automate เต็ม / Human-in-loop / ทำมือ** แล้วผลกรรมเล่นออกมาทันที สอนว่า "AI ลงตรงไหน" + governance/PDPA

**โทนรวม:** dashboard / control-room — มืออาชีพ เชิงกลยุทธ์ น่าเชื่อถือแบบงานธนาคาร แต่ยังเป็นเกมที่ลุ้น (ต่างจาก Sentiment ที่อบอุ่น)

---

## จอที่ต้องออกแบบ (3 จอ)

### จอ 1 · ผู้เล่น (มือถือ, portrait) — หัวใจของเกม
**หน้าที่:** อ่านสถานการณ์ → เลือก 1 ใน 3 → เห็นผลกรรม + มิเตอร์ขยับ

**องค์ประกอบ:**
- **การ์ดสถานการณ์** บนสุด: หัวข้องาน + แท็ก attribute เล็กๆ (ปริมาณ / เสี่ยง / ถูกกำกับ)
  ตัวอย่าง: *"อนุมัติสินเชื่อบุคคล"* แท็ก: เสี่ยงสูง · ถูกกำกับ
- **ปุ่มเลือก 3 ปุ่มใหญ่** แยกกันชัด: `Automate เต็ม` · `Human-in-loop` · `ทำมือ`
- **มิเตอร์ 3 ตัว** (แท่ง/เกจ) ที่ขยับทุกครั้งที่เลือก:
  - ⚡ เร็ว/ประหยัด (สูง = ดี)
  - ⚠️ ความเสี่ยง (สูง = แย่)
  - 🛡️ Compliance (ต่ำ = อันตราย)
- **การ์ด consequence** เด้งหลังเลือก: บรรทัดเดียวบอกผลของตัวเลือกนั้น
- **การ์ด breach** (เฉพาะเลือกพลาดงานเสี่ยงสูง): สีแดง สะดุดตา เช่น "อนุมัติเองไม่มีคน → ผิดเกณฑ์กำกับ อธิบายกับผู้ตรวจไม่ได้"

**States:**
- ยังไม่เลือก — 3 ปุ่มพร้อมกด
- เลือกแล้ว — ปุ่มที่เลือก highlight + consequence + มิเตอร์ animate (+ breach ถ้าพลาด)
- รอ host ไปข้อถัดไป / ปุ่ม "ถัดไป"
- จบเกม — **report card ส่วนตัว**: โปรไฟล์ efficiency vs risk + อันดับ

### จอ 2 · projector (จอกว้าง)
**หน้าที่:** โชว์ว่าทั้งห้องเลือกอะไร — ตัวนี้คือเชื้อเพลิงคุย
- **แท่งรวมต่อข้อ:** สัดส่วน % ที่ห้องเลือก Automate / HITL / Manual (segmented bar)
- **ไฮไลต์ข้อที่ห้องเห็นต่าง** (สัดส่วนใกล้กัน) ให้เด่น — นี่คือจุดที่ facilitator หยิบมาถก
- **scoreboard ทีม** ด้านข้าง (คะแนนรวมเข้า squad)
- สะท้อนหัวข้อสถานการณ์ปัจจุบัน

### จอ 3 · host (แถบควบคุม)
- ปุ่ม "ข้อถัดไป"
- ปุ่ม "เปิดผลรวมทั้งห้อง"
- ตัวเลข: ข้อที่เท่าไร / ตอบแล้วกี่คน

---

## ทิศทางภาพ
- dashboard สะอาด มืออาชีพ น่าเชื่อถือแบบ fintech
- มิเตอร์ 3 ตัว = signature visual ออกแบบให้อ่านสถานะรวดเดียว (เขียว = ดี, เหลือง/แดง = เสี่ยง/ละเมิด)
- การ์ด breach = สีแดงเตือน สะดุด ให้รู้สึก "พลาดแล้ว" จริงๆ
- ปุ่ม 3 ตัวเลือกต้องแยกกันชัด กดง่ายบนมือถือ

## interaction / motion
- เลือก → มิเตอร์ animate ขยับนุ่ม + consequence slide-in
- breach → เด้งเด่นกว่าปกติ (สั่น/flash เบาๆ)
- เปลี่ยนข้อ → การ์ดใหม่ slide เข้ามา

## responsive
- จอ 1 mobile portrait เป็นหลัก
- จอ 2 projector landscape แท่งรวม + scoreboard
- จอ 3 laptop/มือถือ host

## dynamic data
- 6–8 สถานการณ์ (Design ใช้ตัวอย่าง 2–3 อันพอ: "อนุมัติสินเชื่อ", "ตอบ FAQ 500 เคส/วัน", "ตรวจจับธุรกรรมต้องสงสัย")
- มิเตอร์ค่า 0–100 · แท่งรวม % real · scoreboard ตัวเลขจริง

# 🔧 สำหรับ Claude Code (ตัดออกตอนวางใน Design ได้)

**ขั้นตอน:** ใช้ `contract.ts` เดิม → เอา UI จาก Claude Design + prompt ด้านล่างเข้า Claude Code

**Code prompt:**
> สร้างไฟล์ React เดียว `AutomateOrNot.jsx` รับ `GameAPI`; ฝัง array `scenarios` ตามสกีมาใน ContentPack §B.3 — `{id, text, attrs, choices:{automate|hitl|manual:{efficiency, risk, compliance, breach, consequence}}, best}`; state local เก็บ choices + มิเตอร์ส่วนตัว (สะสม delta) + aggregate; ให้คะแนนผ่าน `award()` เมื่อ `choice===best`; ถ้า `breach===true` เด้งการ์ดเตือนแดง; รองรับ `?view=play|screen|host`; โหมด screen โชว์แท่งรวม %/ข้อ + ไฮไลต์ข้อที่ห้องเห็นต่าง + scoreboard; host advance ข้อ + เปิด aggregate; บังคับ 1 คำตอบ/ข้อ/คน; ส่งผ่าน `emit('decision', {scenarioId, choice})` ไม่เรียก backend ตรง; จบเกมโชว์ report card ส่วนตัว (efficiency vs risk)

**จุดพังที่ต้องกันในโค้ด:**
- scenario ต้องมี tradeoff ทุกข้อ (มีตัวเลือก HITL ที่ป้องกันได้ — ContentPack จัดให้แล้ว) ไม่ให้กลายเป็น quiz
- **ต้องเก็บ aggregate ต่อข้อไว้โชว์** — นี่คือของจริงของเกม ไม่ใช่คะแนน
- การ์ด breach ต้องเด่นพอให้จำได้
- normalize มิเตอร์ 0–100 ไม่ให้ทะลุ

**integrate เข้า hub ทีหลัง:** สลับ `emit`/`award`/`station` เป็น Supabase (`decisions` table + `useStation()`) ตาม HUB-SCAFFOLD — ตัวเกมไม่ต้องแก้ · aggregate มาจาก realtime subscribe บน `decisions`
